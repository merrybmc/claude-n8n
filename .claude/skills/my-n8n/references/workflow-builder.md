# Workflow Builder 모듈

워크플로우의 **생성 / 수정 / 분석 / 개선** 요청을 처리합니다.

## 6단계 빌드 프로세스

워크플로우를 새로 만들거나 수정할 때는 아래 6단계를 순서대로 진행합니다.

### 1단계: 요청 분석
- 사용자가 자동화하려는 시나리오를 명확히 이해합니다.
- 트리거(시작 조건), 입력 데이터, 처리 단계, 출력 결과를 분리해서 정리합니다.
- 모호한 부분이 있다면 사용자에게 질문하여 요구사항을 확정합니다.

### 2단계: 노드 검색
- 각 단계에 필요한 노드를 `search_nodes`로 검색합니다.
- 후보 노드를 `get node`로 조회하여 스펙(필수 파라미터, 자격 증명 필요 여부)을 합니다.
- 불확실하면 `tools_documentation`도 함께 참조합니다.

### 3단계: JSON 구성
- 노드 객체와 connections 객체를 포함하는 워크플로우 JSON을 작성합니다.
- 노드 배치 규칙을 따릅니다.
- 각 노드에는 한국어 `notes`를 추가하여 의도를 설명합니다.

### 4단계: 검증
- `validate_workflow`로 JSON 구조를 검증합니다.
- 사용된 각 노드는 `validate_node`로 추가 확인합니다.
- 검증에서 오류가 발견되면 3단계로 돌아가 수정합니다.

### 5단계: 생성/테스트
- 새 워크플로우는 `n8n_create_workflow`로 생성합니다.
- 기존 워크플로우 수정 시에는 먼저 `n8n_workflow_versions`로 현재 버전을 기록한 후 `n8n_update_workflow`를 (또는 partial update) 사용합니다.
- 가능하면 `n8n_execute_workflow`로 실제 동작을 테스트합니다.

### 6단계: 결과 보고
- 생성된 워크플로우 ID, 노드 구성, 동작 흐름을 요약합니다.
- 다음 단계 제안(예: 자격 증명 설정, 실제 데이터 연결)도 함께 제공합니다.

### 노드 배치 규칙

- **방향**: 왼쪽에서 오른쪽으로 데이터 흐름이 진행되도록 배치합니다.
- **간격**: 노드 사이에 X 좌표 간격은 **300PX**로 통일합니다.
- **시작점**: 첫 번째 노드(트리거)는 `[240, 300]` 위치에 둡니다.
- **예시**:
  - Trigger: `[240, 300]`
  - 두 번째 노드: `[540, 300]`
  - 세 번째 노드: `[840, 300]`
  - 분기(branch)가 있을 경우 Y좌표를 `±200px` 정도 어긋나게 배치합니다.

## 안전장치

### 생성 전 검증

- `validate_workflow`로 JSON 구조를 반드시 검증합니다.
- 검증 실패 시 `n8n_autofix_workflow`로 자동 수정을 **1회** 시도합니다.
- 자동 수정 후에도 검증 실패가 지속되면 3단계로 돌아가 수동으로 수정합니다.

### 생성 후 테스트

- `n8n_test_workflow`로 실제 동작 여부를 확인합니다.
- 테스트 결과를 사용자에게 보고하고 문제가 있으면 수정합니다.

### 에러 대응

| 에러 유형 | 대응 방법 |
|---|---|
| API 호출 실패 | fallback 메시지 전송 (원본 데이터 없이도 알림 발송) |
| RSS 파싱 에러 | 해당 항목을 skip하고 나머지 항목으로 계속 진행 |
| AI 요약 실패 | 요약 없이 raw data를 그대로 전송 |

### 결과 리포트

- **성공 시**: 워크플로우 URL과 노드 구성(트리거 → 처리 → 출력 흐름)을 보고합니다.
- **실패 시**: 에러 원인과 구체적인 해결 방법을 함께 보고합니다.

### 안전 원칙

- **삭제** (`n8n_delete_workflow`)와 **전체 수정** (`n8n_update_full_workflow`)은 반드시 사용자 확인 후 실행합니다.
- 부분 수정(`n8n_update_partial_workflow`)은 변경 범위를 사전에 명시한 후 진행합니다.

## 주의사항

- 자격 증명(credentials)이 필요한 노드는 사용자에게 사전에 안내합니다.
- 노드 삭제, 연결 해제 등의 행동은 반드시 사용자에게 확인 후 수행합니다.
- API 키, 토큰 등의 secret data는 절대 워크플로우 본문이나 출력에 노출하지 않습니다.
